# Data model

Everything MemStrata stores lives in a single SQLite database at
`~/.memstrata/core.db` (override via `ML_DB_PATH` or `ML_DATA_DIR`).
There are six base tables plus a handful of indexes plus the
`sqlite-vec` virtual table that holds the embeddings.

This document describes each table, what's in it, and why. If you're
looking for the SQL itself, it's all in
[`memstrata/layer3/_db.py`](../memstrata/layer3/_db.py).

---

## Connection lifecycle

Each HTTP request gets its own short-lived `sqlite3.Connection`,
opened via the `get_conn` dependency in
`memstrata/layer3/_db.py` and closed when the request returns.
Background workers (the embedding worker, the pricing sync, the
ingestion service) open their own connections and close them
explicitly.

We deliberately avoid long-lived shared connections because SQLite's
WAL mode degrades when one writer holds open a transaction
indefinitely. Short connections with `journal_mode=WAL` give us
concurrent reads alongside writes without the writer-blocking-readers
behaviour of the default `journal_mode=DELETE`.

The `sqlite-vec` extension is loaded on every new connection via
`_load_vec_extension`. If the extension isn't available on the host
(e.g., the user installed via pip without sqlite-vec's compiled
artifacts), the daemon logs a warning at startup and disables vector
search; everything else still works.

## Tables

### `sessions`

The narrow "session start/end" tracker. One row per `POST /sessions`
call. Has been less-used since chat capture moved to
`chat_sessions` (V5.4 migration 011); kept for backwards
compatibility with older callers.

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | ULID generated by `_db.new_id()` |
| `project_id` | TEXT | The project this session belongs to |
| `started_at` | TEXT | ISO-8601 timestamp |
| `closed_at` | TEXT | ISO-8601 timestamp; NULL until `POST /sessions/{id}/close` lands |
| `context_hash` | TEXT | Hash of the context block that was injected for this session (when the caller supplied one) |
| `client_id` | TEXT | Free-form caller identifier; older callers used "harness" / "browser_ext" before `client_source` arrived on the telemetry table |

---

### `chat_sessions`

Hierarchical per-provider chat session storage, added in V5.4. This is
the table the dashboard's session list reads. Each row corresponds to
one ChatGPT/Claude/Gemini/etc. URL session that the user has interacted
with.

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | ULID generated by `_db.new_id()` |
| `provider_id` | TEXT NOT NULL | `openai`, `anthropic`, `google`, `deepseek`, `xai`, `mistral`, … |
| `external_session_id` | TEXT NOT NULL | The provider's session hash from the URL (e.g. ChatGPT's `c/<uuid>`) |
| `title` | TEXT | Inferred from the first user prompt of the session |
| `first_seen` | TEXT | ISO-8601 timestamp, set on first turn |
| `last_seen` | TEXT | ISO-8601 timestamp, bumped on every subsequent turn |
| `turn_count` | INTEGER | Number of `telemetry_session_timeline` rows linked here |

`UNIQUE (provider_id, external_session_id)` so an upsert path can rely
on conflict resolution. Two indexes: `idx_chat_sessions_provider` and
`idx_chat_sessions_last_seen DESC`.

---

### `telemetry_session_timeline`

The main fact table. One row per turn. Combines harness-side metadata
(token counts, baseline measurements) with browser-extension content
capture (role, text, char_count).

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK AUTOINCREMENT | |
| `session_id` | TEXT NOT NULL | The session this turn belongs to. References `sessions.id` for harness callers; arbitrary for browser-ext callers. |
| `message_id` | TEXT | Dedup key for stream-pause UPSERT. Unique per DOM node per session when set. |
| `client_source` | TEXT | `'browser_ext'`, `'harness'`, `'coding'`, or NULL for legacy callers. Used by the dashboard to split Chat vs Coding totals. |
| `turn_id` | INTEGER NOT NULL | Per-session turn counter, monotonic |
| `project_id` | TEXT NOT NULL | The project this turn was captured under |
| `user_id` | TEXT | Optional caller-supplied user identifier |
| `provider` | TEXT | LLM provider name |
| `model` | TEXT | Model identifier as the caller reports it |
| `actual_input_tokens` | INTEGER | Counted post-injection |
| `actual_output_tokens` | INTEGER | Counted from the response |
| `saved_input_cost_usd` | REAL | Computed in `memstrata.layer3.pricing.lookup.compute_input_savings_usd` |
| `saved_cache_cost_usd` | REAL | Same module, KV-cache savings |
| `saved_output_cost_usd` | REAL | Same module; non-zero only when a cohort baseline has closed (Pro tier) |
| `measurement_basis` | TEXT | `'input_measured'`, `'cache_measured'`, `'output_cohort_measured'`, …. Provenance string |
| `baseline_period` | INTEGER | 1 when the row was captured during a cohort-measurement window (Pro tier) |
| `recorded_at` | TEXT | ISO-8601 timestamp |
| `chat_session_id` | TEXT | FK → `chat_sessions.id`. NULL for harness/IDE-source turns. |
| `role` | TEXT | `'user'`, `'assistant'`, `'system'` |
| `text` | TEXT | Turn content as captured by the extension. NULL for harness-source turns. |
| `char_count` | INTEGER | Length of `text` when present |
| `baseline_no_context` | INTEGER | Smallest naive baseline the caller computed (prompt only) |
| `baseline_full_repo` | INTEGER | Largest naive baseline (full repo dump) — used by the savings math |
| `baseline_naive_rag` | INTEGER | Middle baseline (naive RAG without injection) |
| `injected` | INTEGER | 0/1: did the caller inject context for this turn? |
| `cache_hit_estimated` | INTEGER | 0/1: did the caller infer a KV-cache hit? |

Indexed by `chat_session_id`. A composite unique index on
`(session_id, message_id)` exists when `message_id` is non-NULL, used
by the upsert path for stream-pause dedup.

### `rationale_log`

Free-form notes attached to a session or turn. Used by the harness
(in the Pro tier) to record why it made a particular injection
decision; used by other callers as a debug breadcrumb.

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK AUTOINCREMENT | |
| `session_id` | TEXT | Optional |
| `project_id` | TEXT | Optional |
| `created_at` | TEXT | ISO-8601 timestamp |
| `content` | TEXT | The note |
| `chat_session_id` | TEXT | FK → `chat_sessions.id`; NULL when associated with a non-chat session |

Indexed by `chat_session_id`.

---

### `provider_pricing`

The materialized view of upstream model rates. Populated by
`openrouter_sync.sync_loop` once per 24 hours; falls back to
`pricing/pricing_matrix.json` when the network is unavailable.

| Column | Type | Notes |
|---|---|---|
| `provider` | TEXT NOT NULL | |
| `model` | TEXT NOT NULL | |
| `input_per_m` | REAL NOT NULL | USD per million input tokens |
| `output_per_m` | REAL NOT NULL | USD per million output tokens |
| `cache_write_per_m` | REAL | USD per million cache-write tokens (Anthropic-style cache pricing) |
| `cache_read_per_m` | REAL | USD per million cache-read tokens |
| `fetched_at` | TEXT NOT NULL | ISO-8601 when this row was last updated |
| `PRIMARY KEY (provider, model)` | | UPSERT on conflict |

The pricing matrix doesn't expire: a stale row is preferable to no
row at all when the dashboard wants to compute savings.

---

### `settings`

Generic key-value store. Used today for the `current_plan` setting
(populated by the Pro overlay, read by the dashboard) and any future
runtime settings that don't deserve their own table.

| Column | Type | Notes |
|---|---|---|
| `key` | TEXT PK | |
| `value` | TEXT NOT NULL | Stored as TEXT regardless of semantic type — parse on read |

## Vector storage (`sqlite-vec`)

When `sqlite-vec` is available, the daemon creates a virtual table
that holds the embeddings of captured chat turns. The schema is
created lazily by the embedding worker on first write — the migration
SQL isn't in `_db.py` because the vec0 extension's syntax differs
slightly from standard SQLite DDL.

The current shape (subject to change as `sqlite-vec` evolves):

```sql
CREATE VIRTUAL TABLE telemetry_timeline_vec USING vec0(
    embedding float[768]
);
```

A separate `embeddings` table maps `telemetry_session_timeline.id`
to the corresponding row in `telemetry_timeline_vec`. Search queries
join the two and rank by cosine distance.

The retrieval layer (`memstrata/layer3/retrieval.py`) is the only
consumer; everything else reads through it.

## Migrations

We don't yet have a formal migration framework. New columns are added
via `ALTER TABLE … ADD COLUMN` calls in `_db.py::_migrate()`, which
runs on every startup. Adding a column is idempotent (we catch
`OperationalError` when the column already exists); dropping or
renaming columns isn't supported and would require a manual data
migration step.

If you need a non-trivial migration:
1. Add the SQL to `migrations/0XX_description.sql`
2. Call it from `init_db` after the base schema executes
3. Wrap the call in a try/except that logs the failure but keeps
   the daemon alive — [Rule 5 (defensive everywhere)](hard-rules.md) applies
   to migrations too

## What's NOT in the Open schema

The Pro tier adds two tables via `memstrata_pro.pro_schema`:
- `plan_features` — the JSON-encoded feature flag list per plan tier
- `stripe_customers` — Stripe customer ID → internal user ID mapping
- A `current_plan` row seed in `settings`

Plus a `cohort_baseline` table created lazily by the Pro cohort module
(`memstrata_pro.baseline.cohort.ensure_table`). The Open daemon
never touches that table; the `_NoOpCohortApi` in `api_server.py`
returns "no baseline" for every query.

If you find yourself wanting to add a column to the Open schema for
billing or plan-tier purposes, that's the signal that the work
belongs in the Pro tier instead.
